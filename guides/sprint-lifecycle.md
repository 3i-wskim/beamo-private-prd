# Sprint Lifecycle — 스프린트 사이클 라이프사이클

> [ai-pipeline-guide.md](./ai-pipeline-guide.md) §4 스프린트 진행의 상세 문서

---

## 사이클 구성

하나의 스프린트는 **여러 사이클**로 구성될 수 있다.

```
스프린트 = 설계 기간 + 사이클 1 + [사이클 2] + [사이클 3] + ...
```

### 각 사이클의 4단계

```
워킹 데이 ──▶ 블로커 머지 데이 ──▶ 논의 & 반영 ──▶ 다음 사이클 결정
                + 논제 정리                        │         │
                                                 계속      완료
                                                  │         │
                                            (워킹 데이로)  프로젝트 종료
```

---

## 1단계: 워킹 데이

### 기간 산정

- **개발자가 PRD 기반으로 산정** (기획자가 정하지 않음)
- 개발자가 여러 명이면 개발자 간 협의
- 소통 형식: "1주 기반 2사이클", "2주 기반 3사이클"

### 진행 규칙

```
[코딩중] ──── 블로커 발생 ──▶ [블로커 작성]
   ▲                              │
   │                         계속 진행 가능?
   │                          │         │
   │                         Yes        No
   │                          │         │
   └──── 작업 계속 ◀──────────┘    [워킹데이 종료]
   │
   └──── 작업 전부 끝남 ──▶ [완료 브랜치 병합] ──▶ [블로커 커밋]
```

### 커밋 정책

| 규칙 | 설명 |
|---|---|
| 워킹 데이 중 커밋 | 가능하지만 **완료 브랜치에는 넣지 않음** |
| 블로커 커밋 | 워킹데이 종료 시 **한 번에** 커밋 권장 |
| 권장 이유 | 블로커 추가/수정으로 커밋 히스토리가 지저분해지는 것 방지 |
| 임시 보관 | 개인 프로젝트에 블로커 임시 작성 → 완료 시 이 프로젝트에 추가 |

### 전원 병합 완료 시

```
개발자 A 병합 ──┐
개발자 B 병합 ──┤──▶ 전원 병합 완료? ──▶ Yes ──▶ 마지막 병합 개발자가
개발자 C 병합 ──┘                                기획자에게 Google Chat 노티
                                        No ──▶ 대기
```

---

## 2단계: 블로커 머지 데이 + 논제 정리

- 전원의 블로커를 스프린트 브랜치에 통합
- 블로커 내 이슈를 분류:

| 이슈 유형 | 처리 주체 | 처리 방식 |
|---|---|---|
| 기획 이슈 | 기획자 | 직접 결정 |
| 디자인 이슈 | 기획자 + 디자이너 | 협업 |
| 기술 이슈 | 개발자 간 | 논의 |
| 복합/대규모 이슈 | 전원 | 전체 회의 |

---

## 3단계: 논의 & 반영

### 소규모 이슈

- 기획자/디자이너가 비동기로 해결
- 결정 사항을 requirement + PRD에 직접 반영

### 대규모 이슈

```
논제 정리 (회의 주관자)
     │
     ▼
전체 회의
     │
     ▼
결정 사항 도출
     │
     ▼
AI로 requirement + PRD 반영
     │
     ▼
PRD v(N+1) 태그
```

### ADR 작성 (선택)

논의 & 반영 과정에서 기획자가 판단하여 ADR을 작성할 수 있다.
모든 결정을 기록하는 것이 아니라, **히스토리로 남기지 않으면 레거시 문제가 생길 수 있는 건**에 한해서만 작성한다.
상식적인 부분이나 간략한 논의로 끝나는 건은 기록하지 않는다.

작성 가이드: [adr-writing-guide.md](./adr-writing-guide.md)

---

## 4단계: 다음 사이클 결정

| 결정 | 조건 | 다음 행동 |
|---|---|---|
| **다음 사이클 진행** | 미해결 작업 또는 추가 요구사항 존재 | PRD 버전 갱신 → 워킹 데이 재시작 |
| **범위 축소 후 완료** | 남은 작업을 다음 스프린트로 이관 | 현재 범위까지만 마무리 |
| **프로젝트 완료** | 모든 requirement 충족 | 최종 태그 + 문서 정리 |

---

## 이상적 흐름 vs 현실

```
이상적:  PRD v1 → 워킹데이 → 완료
현  실:  PRD v1 → 워킹데이 → 블로커 → v2 → 워킹데이 → 블로커 → v3 → 범위축소 → 완료
```

> "2번처럼 깔끔하게 끝나는 경우는 흔치 않다. 보통 스프린트 설계 당시에 발견하지 못한 문제가 발생될 가능성이 매우 크다." — ai-pipeline-guide 원문
