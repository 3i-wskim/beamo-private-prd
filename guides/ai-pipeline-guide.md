## 2. 프로젝트 설계

> 기획 → 개별 PRD 작성 → 병합 → 확정의 과정

### 설계 단계 흐름

```
① 기획자가 Requirement 작성 (initiatives/{slug}/requirement/)
         │
         ▼
② 스프린트 설계 기간 시작
         │
         ▼
③ 각 작업자: 개별 브랜치 생성 + prd-{작업자명}/ 폴더에 AI로 PRD 작성
         │
         ▼
④ 작업자별 PRD 완료 → 스프린트 브랜치에 병합
         │
         ▼
⑤ 전원 병합 완료
         │
         ▼
⑥ 회의 주관자: AI로 PRD 통합 → 의견 차이 항목 정리
         │
         ▼
⑦ 설계 리뷰 회의 → 차이점 논의 및 결정
         │
         ▼
⑧ AI로 requirement + PRD 최종 반영 → 커밋/푸시 → PRD v1 확정
```

### 단계별 설명

| # | 단계 | 누가 | 무엇을 | 핵심 규칙 |
|---|---|---|---|---|
| ① | Requirement 작성 | 기획자 | 새로운 과업을 initiative에 추가 | `requirement/` 폴더에 SO-XX 형식 |
| ② | 설계 기간 시작 | 전원 | 스프린트 설계 기간 진입 | — |
| ③ | 개별 PRD 작성 | 각 작업자 (개발자 + 디자이너) | 자신의 AI로 requirement 기반 PRD 작성 | **특정 의존성 금지** — 기획 문서만 보고 각자 독립 작성 |
| ④ | 스프린트 브랜치 병합 | 각 작업자 | 본인 작업 완료 → 스프린트 브랜치에 merge | 본인 완료를 팀에 공유하는 행위 |
| ⑤ | 전원 병합 완료 | — | 모든 작업자 병합 확인 | — |
| ⑥ | PRD 통합 준비 | 회의 주관자 | AI로 개별 PRD들을 하나로 병합 + 의견 차이 정리 | "A는 이렇게, B는 저렇게 → 어떻게 할까?" 형식 |
| ⑦ | 설계 리뷰 회의 | 전원 | 차이점을 회의에서 결정 | 모두가 있는 자리에서 통합 |
| ⑧ | PRD v1 확정 | 회의 주관자 | AI로 requirement + PRD 최종 수정 → 커밋/푸시 | 태그: `sprint-{작업명}-v1` |

### + 버튼 플로우 (FR-03 참고)

```
User taps + ──▶ Picker: Tag or Folder?
                    │           │
                    ▼           ▼
               Tag 생성     Folder 생성
                    │           │
                    └─────┬─────┘
                          │
                    Online? ──▶ Yes ──▶ 서버에 즉시 생성
                          │
                          └──▶ No ──▶ 로컬 저장 → upload queue 추가
                                       (pending-upload indicator)
```

### 핵심 원칙

> **AI가 만든 PRD는 작업자가 반드시 숙지해야 한다.**
> PRD는 AI의 산출물이지만, 그 내용을 이해하고 설명할 수 있어야 하는 것은 작업자 본인의 책임이다.

> **개별 PRD에 특정 의존성을 넣지 않는다.**
> 기획자의 requirement를 각자의 AI가 이해한 기반으로 작성한다. 다른 작업자의 PRD를 참조하거나 의존하지 않는다.

## 3. 프로젝트 운영

### 사이즈 산정

프로젝트 규모 소통 방식:

```
"1주 기반 2사이클" = 워킹데이 1주 × 2번 반복
"2주 기반 3사이클" = 워킹데이 2주 × 3번 반복
```

### 사이클 흐름

```
PRD v1 확정
      │
      ▼
┌─────────────────────────┐
│  워킹 데이               │
│  (PRD 기반 AI 기반 구현)   │
└────────────┬────────────┘
             │
        이슈 발생?
        │        │
       Yes       No ──▶ (계속 코딩)
        │
        ▼
  블로커 작성
        │
   계속 진행 가능?
   │            │
  Yes           No
   │            │
   ▼            ▼
(코딩 계속)   워킹 데이 종료
                │
      ┌─────────┤
      │         │
      ▼         ▼
  작업 완료   블로커만 남음
      │         │
      └────┬────┘
           │
           ▼
    블로커 머지 데이
           │
           ▼
  기획자: 기획 문서 점검
           │
           ▼
    이슈 분류 & 처리
    ┌──────┼──────┐
    │      │      │
    ▼      ▼      ▼
  기획   디자인  논의 필요
  이슈   이슈   논제 정리
  기획자  기획자+ → 전체 회의
  결정   디자이너
    │      │      │
    └──────┼──────┘
           │
           ▼
    다음 사이클 필요?
     │           │
    Yes          No
     │           │
     ▼           ▼
  PRD v2 갱신  프로젝트 완료
  다음 사이클   (또는 범위 축소)
```

### PRD 버전 관리

| 시점 | PRD 버전 | 태그 |
|---|---|---|
| 스프린트 설계 완료 | v1 | `sprint-{작업명}-v1` |
| 1차 사이클 후 갱신 | v2 | `sprint-{작업명}-v2` |
| 2차 사이클 후 갱신 | v3 | `sprint-{작업명}-v3` |
| ... | ... | ... |

태그 형식: `sprint-{작업명1}-{작업명2}-v{N}`
- 작업명이 여러 단어면 하이픈으로 연결 (띄어쓰기 대용)
- 예: `sprint-survey-overview-mobile-v1`

### 이슈 분류와 처리 주체

```
블로커 ──▶ 이슈 유형?
               │
     ┌─────────┼─────────┬─────────┐
     │         │         │         │
     ▼         ▼         ▼         ▼
  기획 이슈  디자인 이슈 기술 이슈  복합 이슈
     │         │         │         │
     ▼         ▼         ▼         ▼
  기획자     기획자 ↔   개발자 간  논제 정리
  결정       디자이너   논의       → 전체 회의
```

## 4. 스프린트 진행

### 스프린트 구성 요소

```
┌──────────────────────────────────────────────────────────────┐
│                     하나의 사이클                              │
├────────────┬───────────────┬──────────────┬─────────────────┤
│ 워킹 데이   │ 블로커 머지 데이  │ 논의 & 반영   │ 다음 사이클 결정 │
│            │ + 논제 정리    │              │                 │
│ PRD 기반    │ 블로커 통합    │ 회의 또는     │ 계속 / 범위축소  │
│ AI 기반 구현  │              │ 비동기 결정    │ / 완료          │
└────────────┴───────────────┴──────────────┴─────────────────┘
```

### 워킹 데이 상세 흐름

```
워킹 데이 시작 (기간: 개발자가 PRD 기반으로 산정)
      │
      ▼
┌──▶ PRD 기반 AI 기반 구현
│     │
│     ▼
│   블로커 발생?
│    │       │
│   Yes      No ──▶ (계속 코딩)
│    │
│    ▼
│   블로커 작성 (개인 프로젝트에 임시 보관 가능)
│    │
│    ▼
│   더 진행할 작업 있음?
│    │       │
│   Yes      No
│    │       │
└────┘       ▼
        워킹 데이 종료
             │
             ├── 작업 완료: 완료 브랜치에 병합
             │
             └── 공통: 블로커를 이 프로젝트에 추가해서 커밋
```

### 워킹 데이 규칙

| 규칙 | 설명 |
|---|---|
| **기간 산정** | 개발자가 PRD 기반으로 유추. 여러 명이면 개발자 간 협의 |
| **블로커 처리** | 발생 즉시 블로커 작성. 워킹데이 중에는 개인 프로젝트에 임시 보관 가능 |
| **커밋 정책** | 커밋은 가능하나 **완료 브랜치에는 넣지 않음**. 되도록 **마지막에 한 번만** 커밋 권장 |
| **커밋 권장 이유** | 블로커 추가/수정으로 커밋 히스토리가 복잡해지는 것 방지 |
| **워킹 데이 종료 조건** | PRD 작업 완료, 또는 블로커로 더이상 진행 불가 |
| **전원 병합 완료 시** | 마지막 병합한 개발자가 기획자에게 노티 (Google Chat 자동 발송 권장) |

### 블로커 머지 이후

```
전원 블로커 머지 완료
      │
      ▼
기획자: 기획 문서 점검
      │
      ▼
블로커 내 이슈 분류
      │
      ├── 소규모 ──▶ 기획자/디자이너 비동기 해결 ──┐
      │                                           │
      └── 대규모 ──▶ 논제 정리 → 전체 회의 ────────┤
                                                   │
                                                   ▼
                             결정 사항 → requirement + PRD 반영
                                                   │
                                                   ▼
                                            다음 사이클?
                                             │       │
                                            Yes      No
                                             │       │
                                             ▼       ▼
                                       PRD v(N+1)  범위 축소
                                       태그 생성   또는 완료
                                       다음 사이클
```

## 5. 디렉토리 구조

### 새 패러다임 적용 시 이니셔티브 폴더 구조

```
initiatives/{initiative-slug}/
├── requirement/              ← 기획자 소유 (변경 없음)
│   ├── overview.md
│   └── SO-XX-*.md
├── prd/                      ← 통합 PRD (v1 이후, 리뷰 회의 결과물)
│   └── PRD-XX-*.md
├── prd-{작업자명}/            ← [임시] 첫 스프린트 설계 기간 한정
│   └── PRD-XX-*.md               통합 리뷰 후 prd/로 병합되면 삭제
├── blockers/                 ← 블로커 (이슈 기록, 해결 시 삭제)
│   ├── YYYYMMDD-gitid-issue.md
│   └── _pending-delete/     ← 해결 완료 대기 (작성자 확인 후 삭제)
├── adr/                      ← 의사결정 기록 (ADR, 영구 보관)
│   ├── README.md
│   └── ADR-NN-*.md
└── spec-review/              ← 크로스팀 리뷰 교환 (기존과 동일)
```

> **`prd-{작업자명}/`은 임시 폴더다.**
> 첫 스프린트 설계 기간 중에만 존재한다. 설계 리뷰 회의를 거쳐 통합 PRD(`prd/`)가 확정되면 삭제한다.
> 이후 사이클(v2, v3...)에서는 통합 `prd/` 폴더만 갱신한다.

### 현행과의 차이

| 항목 | 현행 | 새 패러다임 |
|---|---|---|
| PRD 폴더 | `prd/` 단일 | `prd-{작업자명}/`(임시) → 리뷰 후 `prd/`로 통합 → 개별 폴더 삭제 |
| PRD 작성 주체 | 개발자 1명 (또는 팀) | 각 작업자가 AI로 독립 작성 |
| PRD 통합 시점 | 없음 (처음부터 단일) | 설계 리뷰 회의 후 AI로 병합 |
| 블로커 커밋 시점 | 수시 커밋 | 워킹데이 종료 시 한 번에 커밋 권장 |

## 7. 역할별 책임

### 기획자

```
Requirement 작성
블로커 우선 확인 → 자체 해결 가능하면 기획 문서에 반영, 불가능하면 담당자와 논의
디자인 이슈 → 디자이너와 협업 (혼자 떠안지 않음)
전체 회의 필요 시 논제 정리
ADR 작성 여부 판단 (히스토리로 남겨야 하는 건에 한해)
```

### 개발자

```
AI로 개별 PRD 작성 (requirement만 보고 독립적으로)
PRD 기반 AI 기반 구현
블로커 작성
워킹데이 기간 산정
개발 코드 전원 병합 완료 시 → 기획자 노티
```

### 디자이너

```
AI로 디자인 관점 PRD 작성
기획자와 디자인 이슈 협업
```

### 회의 주관자

```
AI로 개별 PRD 통합
의견 차이 항목 정리
리뷰 회의 진행
최종 PRD 확정 + 태그
```

### 역할 요약 테이블

| 역할 | 핵심 책임 |
|---|---|

| **회의 주관자** | PRD 통합, 의견 차이 정리, 리뷰 회의 진행, 최종 확정 |
